Create continuously deployed microservices without breaking a sweat

Easily create and deploy modern, containerized apps composed of microservices.

Existing development tools weren’t built with containers in mind

Developers who have started down the microservices path have noted that it is very time consuming to begin building microservices in containers with their existing tools. The learning curve is steep and there are a lot of steps to follow to get things set up and working. Once you get your own development environment working, you have to help teammates go through the same process and learning curve.

Here’s how it usually goes: First, you need to select a software stack that is lightweight enough for microservices in containers. Then you need to pull down all of the components. There are probably some tutorials you’ll need to follow to hook the components of the stack together and some boilerplate code to copy into your IDE.

Next, you need a development environment. Desktop IDEs are traditionally intended to build and test code on the local machine under the local OS, which is often Windows or Mac OS. Code that runs fine on the desktop might not be portable to the target environment, which is Linux containers running in the cloud. Many IDEs can be configured to trigger remote builds, execute code remotely, and connect to remote debuggers, but there is a lot of setup involved and some complexity to manage. Oh, and you also have to deal with keeping the source code synchronized between your desktop and the target machine. You don’t want to have to check code in just to try building it on the target machine.

When it comes to building your project, ideally you’d like to just hit a button in your IDE. But that won’t build into a container automatically. If you wait to move to containers later in the project, you might not be able to adequately test your code and detect problems with the code in front of you, while it’s fresh in your mind.

Also, how many places do you need to configure your build: your IDE, your test/integration environment, your CI/CD server? When one of the components of your stack gets updated because a new, fixed version is released, how many places do you need to touch to make sure all environments get the right version?

Speaking of CI/CD, you need an environment for that, so you need to download Jenkins, figure out where to run it, and climb the learning curve. For CI/CD to be effective, you need to tie it into some of your processes so it gets kicked off when new code is committed, and you might also want to set up an approval process for deployment.

When it comes to deployments, you want to reduce downtime and mitigate risks while rolling out new code. Do you have enough resources, i.e., two identical production environments to be able to do blue-green or canary deployments? Do you have the infrastructure, such as load balancers, to easily route portions of your workload between your production environments?

It’s enough to easily get discouraged. Wouldn’t it be nice if there was a development platform designed with microservices and containers in mind?

Red Hat OpenShift.io, a cloud-native development environment

To help development teams be productive Red Hat created OpenShift.io. OpenShift.io is a free, cloud-native, team development experience for managing software projects. Optimized for creating container-based applications and microservices-based solutions in the cloud, OpenShifto.io is the ideal path to continuous value delivery and digital transformation.

OpenShift.io lets you get started writing actual, functional code for your microservices project right away. When you create a space in OpenShift.io for your project, you choose from a list of quick starts. You can pick a stack appropriate for microservices, like a RESTful HTTP endpoint based on the Vert.x runtime. When you do this, a lot more is happening than just importing some boilerplate code. All of the components of your selected stack are pulled into your space, along with the information to automatically build the whole project into Linux containers. With nothing more than a browser and an internet connection, you now are ready to write the first method of your microservice, build it, test it, and deploy it. Getting to this point could have taken you days or longer if you had to assemble all of the pieces yourself.
